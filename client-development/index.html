Images

Jellyfin Images API

Image endpoint has 3 sets of options. Sending a single side will always result in image in original ratio.

width/height

Do not use! Those argument together won't respect the ratio of image and stretches it. Also it will upscale images.

fillWidth/fillHeight

Best for logos. Logo has defined area e.g. 600x400 and using fillWidth=600&fillHeight=400 behave like object-fit: contain, so no side will be larger as requested size

maxWidth/maxHeight

Good for posters and so on. Images with closely predictable ratio and a area needs to be filled. Similar to object-fit: cover. Logo has defined area e.g. 400x600 and using maxWidth=400&maxHeight=600 and image is 4000x7000 will return a image with size 400x700.

Issue with Logo and max can be the ratio. e.g. 20:1 logo and 600x400 will result in image size of 8000x400 (no upscale).

Quality

This is always a bit complicated depending on display, user and so on. A well sized image will be small, so a high quality is often beneficial. I would never go below 80, except on HighDPI (dpr), more pixel in lower quality will look better compared to 100% quality and upscaled. Small setting with few options can help and user can decide on eyesight and device.

Format

Jellyfin supports multiple formats best for a lot of use cases is webp, because it has transparency. It can be converted from PNG and JPEG. Modern formats like JXL or AVIF are not supported. Always nice if supported formats by client are send as Accept header, so users can use a dynamic proxy and cen get benefit of JXL and so on.

Size

Best calculate the needed image size and also dpr, because in it will add up especially in listing

Cache busting

Jellyfin has tag in API response and it will change after an image gets replaced in Jellyfin. Some use a cache on server side as well, so tag should be used, so URL changes.

Unauthenticated

Images are unauthenticated and can be accedded with UUID, so no reason to sen ApiKey that ends up in access log
